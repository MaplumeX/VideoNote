## Findings

### 1. Async Task Queue for FastAPI

Comparison of four approaches for offloading long-running video processing tasks (1-10 min):

| Feature | Celery | ARQ | FastAPI BackgroundTasks | Dramatiq |
|---|---|---|---|---|
| **Redis Dependency** | Required (or RabbitMQ) | Required | None | Required (or RabbitMQ) |
| **Setup Complexity** | High (celery + broker + result backend + flower) | Low (arq worker + Redis) | Minimal (built-in) | Medium (dramatiq + Redis + dashboard) |
| **Monitoring** | Flower (full-featured) | Basic (arq dashboard) | None | Dramatiq dashboard |
| **Task Persistence** | Yes (via broker) | Yes (via Redis) | No (in-process, lost on restart) | Yes (via broker) |
| **Task Retry** | Built-in (autoretry, max_retries, backoff) | Built-in (max_tries, retry_delay) | Manual | Built-in (max_retries, min_backoff) |
| **Scheduled Tasks** | Yes (celery-beat) | Yes (cron jobs built-in) | No | Yes (periodic middleware) |
| **Async Native** | No (sync + eventlet/gevent) | Yes (asyncio) | Yes (asyncio) | No (sync) |
| **MVP Simplicity** | Overkill | Good fit | Simplest but limited | Medium |

**Analysis for VideoNote MVP:**

- **FastAPI BackgroundTasks**: Simplest to start, but tasks are in-process and lost on server restart. No progress reporting, no retry, no persistence. Suitable only for fire-and-forget tasks under ~30 seconds. NOT recommended for 1-10 min video processing.

- **ARQ** (recommended for MVP): Asyncio-native (fits FastAPI perfectly), requires only Redis, built-in cron support, minimal setup (~50 lines for worker + tasks). The `arq` worker runs as a separate process, communicating via Redis. Task results are queryable. Retry and timeout built-in. `arq` uses `asyncio` throughout, so `await` calls (e.g., to yt-dlp subprocess, ASR API) work naturally.

  Basic pattern:
  ```python
  # worker.py
  from arq import create_pool
  from arq.connections import RedisSettings

  async def process_video(ctx, video_url: str):
      # 1. download + extract subtitles via yt-dlp
      # 2. or extract audio + call ASR
      # 3. call LLM for notes
      # 4. save result
      return {"status": "done", "notes": "..."}

  async def startup(ctx):
      ctx['redis'] = await create_pool(RedisSettings())

  class WorkerSettings:
      functions = [process_video]
      on_startup = startup
      redis_settings = RedisSettings()
  ```

  ```python
  # api.py (FastAPI endpoint)
  from arq import create_pool
  from arq.connections import RedisSettings

  @app.post("/process")
  async def process(request: VideoRequest):
      redis = await create_pool(RedisSettings())
      job = await redis.enqueue_job("process_video", request.url)
      return {"job_id": job.job_id}
  ```

- **Celery**: Industry standard, most mature, but heavy for MVP. Sync-only (needs eventlet/gevent for concurrency). Configuration is verbose. Flower monitoring is excellent but adds another service. Better suited when you need RabbitMQ, complex routing, or large-scale distributed processing.

- **Dramatiq**: Middle ground. Cleaner API than Celery, but sync-only. Good dashboard. Not as widely adopted. Not asyncio-native, so wrapping async code requires `asyncio.run()` inside task functions.

**Recommendation**: ARQ for MVP. Easiest to pair with FastAPI's async model, single Redis dependency, built-in cron, and adequate monitoring for small scale. If scale demands grow significantly, migrating to Celery later is straightforward since the task function logic stays the same.

### 2. yt-dlp Subtitle Extraction

#### YouTube Subtitle Extraction

yt-dlp supports two types of YouTube subtitles:
- **Manual subtitles** (`--write-subs`): Uploaded by the video creator. Available in `info['subtitles']` dict, keyed by language code (e.g., `en`, `zh-Hans`, `ja`).
- **Auto-generated captions** (`--write-auto-subs`): Machine-generated by YouTube. Available in `info['automatic_captions']` dict.

Python API usage:
```python
import yt_dlp

ydl_opts = {
    'writesubtitles': True,      # download manual subs
    'writeautomaticsub': True,    # download auto-generated subs
    'subtitleslangs': ['en', 'zh-Hans', 'zh'],  # target languages
    'subtitlesformat': 'srt',    # format preference: 'srt', 'ass/srt/best'
    'convertsubs': 'srt',        # convert to srt (supports: ass, lrc, srt, vtt)
    'quiet': True,
}

with yt_dlp.YoutubeDL(ydl_opts) as ydl:
    info = ydl.extract_info(url, download=False)
    # Available subtitle languages:
    manual_langs = list(info.get('subtitles', {}).keys())
    auto_langs = list(info.get('automatic_captions', {}).keys())
```

Subtitle formats returned by YouTube:
- Manual: `srv1`, `srv2`, `srv3`, `ttml`, `vtt` (WebVTT)
- Auto-generated: typically `vtt` only
- After `--convert-subs srt`, all are converted to SRT

**Reliability**: YouTube subtitle extraction is generally reliable. Auto-captions are available for most popular-language videos. Manual subs depend on creator. Some videos have neither.

#### Bilibili Subtitle Extraction

yt-dlp has multiple Bilibili extractors:
- `BiliBiliIE` — standard videos (BV/AV URLs)
- `BiliBiliBangumiIE` — bangumi/anime series
- `BiliIntlIE` — international version (bilibili.tv)

**BiliBiliIE subtitle extraction** (source: `yt_dlp/extractor/bilibili.py`, `_get_subtitles` method):

1. **Danmaku (bullet comments)**: Always available as XML from `https://comment.bilibili.com/{cid}.xml`. These are NOT traditional subtitles but scrolling user comments.

2. **CC subtitles**: Retrieved via `https://api.bilibili.com/x/player/wbi/v2` API. Available in `info['data']['subtitle']['subtitles']`. Each entry contains:
   - `subtitle_url`: JSON URL containing the actual subtitle body
   - `lan`: language code (e.g., `ai-zh` for AI-generated Chinese)

3. **Format**: Bilibili CC subtitles are fetched as JSON and converted to SRT via the `json2srt` method:
   ```python
   def json2srt(self, json_data):
       srt_data = ''
       for idx, line in enumerate(json_data.get('body') or []):
           srt_data += (f'{idx + 1}\n'
                        f'{srt_subtitles_timecode(line["from"])} --> {srt_subtitles_timecode(line["to"])}\n'
                        f'{line["content"]}\n\n')
       return srt_data
   ```

4. **Login requirement**: If `data.need_login_subtitle` is True, subtitles require authentication. yt-dlp will warn and skip. You need to pass cookies (`--cookies-from-browser chrome` or `--cookies cookies.txt`) to access login-required subtitles.

**BiliIntlIE subtitle extraction**:
- Uses `/web/v2/subtitle` API endpoint
- Supports ASS and SRT formats directly from URL
- Also supports JSON format with `json2srt` conversion
- Language key is `lang_key` (defaults to `en`)

**Limitations for Bilibili subtitles:**
- Many Bilibili videos have NO CC subtitles at all (creator must enable them)
- AI-generated subtitles (`ai-zh`) are increasingly available but not universal
- Some subtitles require login/cookies
- Danmaku is not suitable as subtitle replacement (it is user commentary, not transcription)
- Bilibili frequently updates its API, causing yt-dlp breakages (common issue with the platform)
- Rate limiting is aggressive on Bilibili's API

**Fallback strategy for Bilibili**: When CC subtitles are unavailable, extract audio and use ASR (see Section 4).

### 3. Audio Extraction from Video via ffmpeg

#### Best Approach: subprocess + ffmpeg

For Python, using `subprocess.run()` with ffmpeg is the most reliable approach. The `ffmpeg-python` package exists but adds a dependency for marginal convenience.

```python
import subprocess
import os

def extract_audio(video_path: str, output_path: str, format: str = "wav") -> str:
    """Extract audio from video file using ffmpeg.

    Args:
        video_path: Path to input video file
        output_path: Path for output audio file
        format: Output format (wav, mp3, opus)
    """
    cmd = [
        "ffmpeg",
        "-i", video_path,
        "-vn",             # no video
        "-acodec", "copy", # copy audio stream if possible
        "-y",              # overwrite output
        output_path,
    ]

    # For WAV (uncompressed, best for ASR):
    if format == "wav":
        cmd = [
            "ffmpeg", "-i", video_path,
            "-vn",                    # no video
            "-acodec", "pcm_s16le",   # 16-bit PCM
            "-ar", "16000",           # 16kHz sample rate (standard for ASR)
            "-ac", "1",               # mono
            "-y", output_path,
        ]
    # For MP3 (compressed, smaller files):
    elif format == "mp3":
        cmd = [
            "ffmpeg", "-i", video_path,
            "-vn", "-acodec", "libmp3lame",
            "-ar", "16000", "-ac", "1",
            "-b:a", "64k",
            "-y", output_path,
        ]
    # For Opus (best compression, but not all ASR APIs accept it):
    elif format == "opus":
        cmd = [
            "ffmpeg", "-i", video_path,
            "-vn", "-acodec", "libopus",
            "-ar", "16000", "-ac", "1",
            "-b:a", "32k",
            "-y", output_path,
        ]

    subprocess.run(cmd, check=True, capture_output=True)
    return output_path
```

#### Recommended Format for ASR APIs

| Format | Codec | Sample Rate | Channels | Size (10min) | ASR API Support |
|---|---|---|---|---|---|
| **WAV** | PCM s16le | 16kHz | Mono | ~19MB | Universal (all APIs accept) |
| MP3 | libmp3lame | 16kHz | Mono | ~4.8MB | Most APIs |
| Opus | libopus | 16kHz | Mono | ~2.4MB | Limited (OpenAI accepts, others vary) |

**Recommendation**: Use **WAV (PCM s16le, 16kHz, mono)** as the default for ASR submission. It is universally accepted, lossless, and the small additional file size is irrelevant since it is a transient file (not stored long-term). For storage or when bandwidth is a concern, MP3 at 64k is a safe fallback.

#### yt-dlp Direct Audio Extraction

yt-dlp can extract audio directly without saving the full video:

```python
ydl_opts = {
    'format': 'bestaudio/best',
    'postprocessors': [{
        'key': 'FFmpegExtractAudio',
        'preferredcodec': 'wav',
    }],
    'outtmpl': '/tmp/%(id)s.%(ext)s',
}
```

This downloads only the audio stream and converts it via ffmpeg. For VideoNote, this is more efficient than downloading the full video then extracting audio.

### 4. Cloud ASR API Comparison

| Feature | OpenAI Whisper API | Google Cloud STT | Alibaba ASR (Paraformer) |
|---|---|---|---|
| **Pricing** | $0.006/min | $0.016/min (standard), $0.024/min (enhanced) | ~0.006 CNY/sec (~0.005 USD/min) for standard |
| **Max Audio Length** | 25 MB per request (files API up to 2h) | ~480 min (standard), ~60 min (enhanced batch) | Up to 500 min (file submission) |
| **Chinese Quality** | Good | Good (enhanced model better) | Excellent (purpose-built for Chinese) |
| **English Quality** | Excellent | Excellent | Good |
| **Mixed ZH+EN** | Good | Good | Excellent (code-switching support) |
| **Streaming** | No (batch only) | Yes (streaming API) | Yes (real-time streaming) |
| **Rate Limits** | 50 RPM (Tier 1), scales with tier | 800-8000 qpm depending on model | Varies by account tier |
| **Output Formats** | Plain text, SRT, VTT, Verbose (timestamps) | Plain text, with word timestamps | Plain text, with timestamps |
| **Speaker Diarization** | No | Yes (enhanced model) | Yes (in some models) |
| **Key Advantage** | Best English, simplest API | Streaming + diarization | Best Chinese, cheap, code-switching |
| **Key Limitation** | No streaming, no diarization, rate limits | Expensive, complex API setup | Weaker English, complex Alibaba Cloud setup |

#### OpenAI Whisper API Details

```python
from openai import OpenAI

client = OpenAI()

# Simple transcription
with open("audio.wav", "rb") as f:
    transcript = client.audio.transcriptions.create(
        model="whisper-1",
        file=f,
        language="zh",       # optional, helps accuracy
        response_format="verbose_json",  # for timestamps
        timestamp_granularities=["segment"],  # or ["word"]
    )
```

- Max file size: 25 MB per request. For longer audio, use the chunked approach or split files.
- The `gpt-4o-transcribe` model is also available (newer, may be better for some use cases).
- Rate limits: starts at 50 RPM for Tier 1, increases with usage tier.

#### Google Cloud Speech-to-Text Details

- V1 API (standard): cheaper, no speaker diarization, up to ~480 min
- V2 API: newer, supports diarization and longer audio
- Requires Google Cloud project + service account
- Streaming API available for real-time use (but VideoNote is batch, so not needed)
- Chirp model (Universal): auto-detects language, good for mixed ZH+EN

#### Alibaba Cloud ASR Details

- **Paraformer-large**: Open-source model available on ModelScope/HuggingFace. Can run locally for free.
- **Alibaba Cloud commercial API**: `alibabacloud-nls` Python SDK. Cheaper for Chinese content.
- **Best for**: Content primarily in Chinese, especially Bilibili videos.
- **Limitation**: Setup requires Alibaba Cloud account, AccessKey, and navigating Chinese-language documentation.

**Recommendation for VideoNote**:
- **Primary**: OpenAI Whisper API — simplest integration, excellent English + good Chinese, reasonable pricing.
- **Fallback for Chinese-heavy content**: Alibaba ASR (Paraformer) — superior Chinese quality and code-switching, cheaper at scale.
- **Self-hosted option**: `funasr` (Paraformer model from ModelScope) — free, runs locally, excellent Chinese. Good for users who want to avoid API costs or rate limits. Requires GPU for real-time but CPU works for batch.

### 5. WebSocket vs SSE for Progress Updates

| Feature | Server-Sent Events (SSE) | WebSocket |
|---|---|---|
| **Direction** | Server → Client only | Bidirectional |
| **Protocol** | HTTP/1.1+ (native) | ws:// (upgrade handshake) |
| **Reconnection** | Auto (browser built-in) | Manual (must implement) |
| **FastAPI Support** | `sse-starlette` package (well-maintained) | Native via `WebSocket` class |
| **Complexity** | Very low | Medium |
| **Proxy/Firewall** | Works through most proxies | Some corporate proxies block ws:// |
| **Browser Support** | All modern browsers | All modern browsers |
| **Use Case Fit** | One-way progress updates | Chat, collaboration, bidirectional |

#### SSE in FastAPI (with sse-starlette)

```python
from sse_starlette.sse import EventSourceResponse
from fastapi import FastAPI, Request

app = FastAPI()

@app.get("/tasks/{job_id}/progress")
async def task_progress(job_id: str, request: Request):
    async def event_generator():
        while True:
            if await request.is_disconnected():
                break
            status = await get_job_status(job_id)  # check Redis/ARQ
            yield {
                "event": "progress",
                "data": json.dumps({
                    "progress": status.progress,
                    "stage": status.stage,
                    "message": status.message,
                }),
            }
            if status.done:
                yield {"event": "complete", "data": json.dumps(status.result)}
                break
            await asyncio.sleep(1)
    return EventSourceResponse(event_generator())
```

Client-side:
```javascript
const es = new EventSource(`/tasks/${jobId}/progress`);
es.addEventListener("progress", (e) => {
    const data = JSON.parse(e.data);
    updateProgressBar(data.progress, data.message);
});
es.addEventListener("complete", (e) => {
    showResult(JSON.parse(e.data));
    es.close();
});
```

#### WebSocket in FastAPI

```python
from fastapi import WebSocket, WebSocketDisconnect

@app.websocket("/ws/tasks/{job_id}")
async def task_progress_ws(websocket: WebSocket, job_id: str):
    await websocket.accept()
    try:
        while True:
            status = await get_job_status(job_id)
            await websocket.send_json({
                "progress": status.progress,
                "stage": status.stage,
            })
            if status.done:
                await websocket.send_json({"status": "complete", "result": status.result})
                break
            await asyncio.sleep(1)
    except WebSocketDisconnect:
        pass
```

**Recommendation for VideoNote**: **SSE** is the better fit because:

1. VideoNote only needs server → client progress updates (no bidirectional communication needed)
2. SSE auto-reconnects when the client reconnects (important for long-running tasks)
3. Simpler implementation on both server and client
4. Works through proxies and load balancers without extra configuration
5. `sse-starlette` integrates cleanly with FastAPI
6. SSE is semantically correct for "progress stream" — it is a one-way event stream

WebSocket would only be preferable if you need the client to send messages during processing (e.g., pause/cancel commands), but for an MVP, a simple `DELETE /tasks/{job_id}` REST endpoint handles cancellation.

### Caveats / Not Found

- No existing application code in the repository yet — this is a greenfield project.
- YouTube API access from the local environment has SSL issues; production deployment will need to handle network configuration.
- Alibaba ASR pricing is approximate and varies by region and service tier; exact pricing requires Alibaba Cloud account.
- ARQ's `arq-dashboard` is a separate package with limited features compared to Celery's Flower; for MVP this is acceptable.
- Bilibili API stability is a known risk; yt-dlp updates frequently to keep up with their API changes. Pinning yt-dlp version and updating regularly is recommended.
